# Impasto session handoff — 2026-07-13

## Shipped state

Impasto **0.7.0** is the current Blender 5.1 build. The resident GPU
brush now keeps channel canvases on the GPU between strokes, performs no
pen-up image readback, rejects hidden surfaces, and stores one atomic GPU undo
record across all affected channels.

The 0.7.0 preview adds image-based studio lighting and a strict common-stack
GPU path: when the active Paint layer is topmost and shares one UV domain, lower
Fill/Paint layers and Kiln's baked normal are included without readback. Mixed
UVs, image masks, or participating upper layers explicitly fall back to the
active-layer preview.

Correction recorded 2026-07-19: lower normal image transport is present, but
normal layers are still combined as encoded RGB with ordinary MIX. An opaque
upper normal canvas consequently replaces the Kiln/lower normal. True layered
normal parity remains open and requires RNM/UDN vector composition.

Post-handoff update: **0.6.0** adds explicit GPU Stroke Opacity, a clearer Layer
Opacity control, and `V` toggling between resident painting and Blender's
authoritative material without exiting/re-entering the GPU operator. Entering
inspection still pays one dirty-region synchronization, but preserves resident
textures and GPU undo history; returning to paint is immediate.

Idle Material Synchronization can perform an automatic handoff after an
adjustable delay, but it defaults off: routine painting remains GPU-resident
and zero-readback. `V` / Inspect Blender Material is the explicit inspection
boundary. Ctrl-S/Ctrl-Shift-S defer saving until resident changes have reached
the Blender Images; menu saves and exporters require the visible Flush for
Save / Export action first.

The live overlay has four persistent display modes:

- **Lit PBR** — an environment-style GGX approximation of the resident Base
  Color, Metallic, Roughness, Tangent Normal, and Height data;
- **Raw Tangent Normal** — encoded normal RGB;
- **Neutral Normal Lighting** — normal and Height relief without material-color
  distractions; and
- **Height Grayscale** — the resident Height field directly.

Mode switching performs no flush, readback, image write, or undo boundary.
Diagnostic modes avoid unnecessary texture reads and PBR work. The shader also
corrects transparent scalar defaults, partial-normal strength, mirrored and
degenerate UV tangent frames, and Height derivatives.

The staggered pinholes previously visible inside projected strokes were traced
to a depth tolerance that claimed to allow a fraction of the local pixel slope
but actually capped the allowance near numerical epsilon. The corrected policy
uses a bounded local depth footprint while retaining discontinuity gates for
rear surfaces, silhouettes, thin foreground features, and occlusion boundaries.

## Editing settings during a resident session

The first attempt relied on `PASS_THROUGH` plus Blender region rectangles. It
failed interactively because an N-panel can overlap the `WINDOW` region and its
press reaches the modal painter before normal UI handling; slider clicks were
therefore reported as dabs. An explicit resident input state is the reliable
solution in 0.5.2:

1. press **P** to pause dab capture;
2. edit color, Metallic, Roughness, normal, Height, Radius, Hardness, or preview
   mode in Blender's UI;
3. press **P** to resume; and
4. resume before RMB/Esc when the session should flush and finish.

Pausing preserves GPU textures and undo history and performs no synchronization.
While paused, every pointer event passes through, including events over the
canvas. A future Blender event-routing API or a non-modal workspace-tool
integration could remove this extra state, but it is not a painting-engine
blocker.

## Why the live view is not exact Blender Material Preview

Blender's material graph samples `bpy.data.images`; the newest resident pixels
exist only in Impasto-owned GPU textures until flush. The add-on therefore draws
a temporary shader over the mesh. This is a major source of the latency
improvement, but it cannot reproduce Blender's exact studio light/HDRI and
Principled evaluation through the public Python/GPU API. Blender's material is
authoritative again after explicit flush or normal session exit.

## Newly agreed roadmap

### Additional paint channels

- Subsurface painting, preserving the native scalar/vector units and Non-Color
  semantics of the relevant Principled inputs.
- Emission Color plus Emission Strength/luminosity painting. Color and strength
  remain distinct so emission is not limited to display-white RGB.

### Cross-channel stencils and texture application

One image stencil should modulate the same logical brushstroke across every
enabled channel while preserving registration between color, scalar PBR,
Tangent Normal, Height, Emission, and Subsurface targets. It should also support
intentional texture application rather than only round procedural dabs.

The user will describe the desired interaction in a later session. Do not invent
or freeze projection, transform, alpha-versus-luminance interpretation, tiling,
or per-channel payload behavior before that description.

### GPU equivalents of Blender brush families

Blender's open source is a useful behavioral specification, but its native
painting implementation is not a drop-in shader library for a Python add-on. It
can use internal C/C++ data structures, image caches, undo machinery, and direct
canvas access unavailable through the public add-on API. Implement compatibility
in tiers:

1. Draw/stamp variants using the current resident dab engine.
2. Adjustable alpha textures, custom falloff, rotation, rake, jitter, spacing,
   and pressure mapping.
3. Destination-sampling Soften/Blur, Smear, and Mask tools, likely using
   ping-pong textures to avoid sampling a render target while writing it.
4. Source/state-heavy Clone, Fill, Gradient, and specialized tools with their
   own interaction and preview models.

Every tier must retain synchronized multi-channel coverage, front-surface
projection, resident atomic undo, and no-readback pen-up. Normal, Height,
scalar, and color targets require channel-specific blend semantics even when
they share one spatial alpha.

## Add-on versus Blender core

A full Blender fork is not currently justified. The add-on can plausibly
deliver the layer stack, GPU Draw/alpha/stencil brushes, extra PBR channels,
destination-sampling tools, cloning, export, and most of the intended workflow.
Continue until a measured public-API wall is reached.

Prefer small, general upstream API patches when deeper integration is needed.
Likely candidates are access to resident material textures, cleaner modal/tool
event routing, grouped native undo, richer brush-asset data, and a supported way
for Material Preview to consume external GPU textures. A maintained fork would
only be warranted if the product requires perfect native brush parity, exact
Material Preview of unflushed data, native undo/tool/workspace integration,
internal paint caches, or other behavior that cannot be exposed incrementally.
