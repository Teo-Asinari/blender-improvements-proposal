# Scale-Aware Remesh Preview and Safety: Technical Research

This note supports [Proposal §5, “Scale-Aware Remesh Preview and Safety”](../PROPOSAL.md#5-scale-aware-remesh-preview-and-safety). Its most important constraint is terminology: Blender's destructive **Voxel Remesh operation** and its non-destructive **Remesh modifier** are separate entry points with different geometry inputs and UI integration requirements.

## Recommendation

Prototype a conservative object-space risk estimate, explicit scale-aware sizing helper, and bounded viewport overlay in an add-on. A Blender-wide safety feature—covering the native Sculpt UI, `R` modal overlay, `Ctrl-R` execution, search/menu/script invocation, and the native modifier panel—requires Blender-core integration.

## Entry points and geometry sources

### Destructive Voxel Remesh operation

`bpy.ops.object.voxel_remesh`, exposed in Sculpt Mode as `Ctrl-R`, destructively replaces the mesh. Its settings live on the mesh datablock, including `Mesh.remesh_voxel_size` and `Mesh.remesh_voxel_adaptivity`. The documented voxel size is an **object-space** distance; lower values produce denser topology.

The operation uses original mesh data, not geometry generated by modifiers, shape-key deformation, or rigging, and does not work with Multiresolution. It can reproject volume, paint masks, face sets, and color attributes, while other mesh data layers are lost. Its estimator should therefore inspect original mesh coordinates.

### Non-destructive Remesh modifier

`RemeshModifier` evaluates non-destructively in the modifier stack. It has legacy `BLOCKS`, `SMOOTH`, and `SHARP` modes controlled by octree depth and scale, plus `VOXEL` mode controlled by voxel size and adaptivity. Legacy resolution is bounding-box-relative; voxel mode uses an absolute local/object-space distance.

Because this path sees geometry produced earlier in the stack, a trustworthy estimate must inspect the evaluated input immediately before the modifier. An estimator based only on `object.data` must be labeled approximate.

The two paths can share arithmetic, but only through an interface that explicitly identifies the coordinate space and geometry source. “Cover every entry point,” as proposed, should mean consistent risk language and interaction goals—not one indiscriminate estimator.

Authoritative references: [Sculpt Remesh manual](https://docs.blender.org/manual/en/dev/sculpt_paint/sculpting/tool_settings/remesh.html), [Remesh modifier manual](https://docs.blender.org/manual/en/dev/modeling/modifiers/generate/remesh.html), [Mesh Python API](https://docs.blender.org/api/5.0/bpy.types.Mesh.html), and [RemeshModifier Python API](https://docs.blender.org/api/current/bpy.types.RemeshModifier.html).

## Defaults and scale semantics

The public Mesh API documents `remesh_voxel_size = 0.1` and `remesh_voxel_adaptivity = 0.0`; adaptivity above zero disables Fix Poles. Historical voxel-mode modifier defaults are also `voxel_size = 0.1` and `adaptivity = 0.0`. These are datablock defaults, not scale-aware recommendations. Modifier-mode and version-specific defaults should be read from RNA at runtime rather than duplicated in add-on constants.

The native voxel-size value must not be silently reinterpreted as world space. If an object has unapplied scale, its apparent world-space cell size changes. Non-uniform scale makes the apparent cells non-uniform in world space. A safe UI should:

- retain and show the exact native object-space value;
- derive world-space axis sizes from the effective transform;
- warn for unapplied or non-uniform scale;
- offer an explicit **Set from World Target** helper with a documented conversion policy;
- never apply object scale automatically as part of remeshing.

A conservative conversion can divide a requested world-space size by the largest effective axis scale, ensuring no transformed axis is coarser than the target. For sheared transforms, singular values are more correct than reading the three scale properties.

## Cheap estimates

For object/local-space axis-aligned bounds `d = (dx, dy, dz)` and voxel size `v > 0`:

```text
nx = ceil(dx / v) + padding
ny = ceil(dy / v) + padding
nz = ceil(dz / v) + padding
bounding_cells = nx * ny * nz
longest_axis_cells = ceil(max(dx, dy, dz) / v)
```

Use a small documented padding only to avoid implying false precision. `bounding_cells` is a conservative domain/risk indicator, **not a prediction of OpenVDB memory**. OpenVDB is sparse; active topology, narrow bands, implementation details, adaptivity, and temporary buffers affect real memory. Do not translate the value to megabytes unless Blender core exposes a stable allocation estimate.

A complementary output-complexity score is:

```text
surface_cells ~= mesh_surface_area / v^2
estimated_faces ~= C * surface_cells
```

`C` depends on topology and remesher version. Until calibrated empirically, show `surface_cells` as a relative score rather than promising a face count. Adaptivity may reduce final faces but does not eliminate voxelization cost, so it should not lower the preflight domain-risk score.

Recommended warning behavior:

- display longest-axis cells because the number is intuitive;
- compare configurable logarithmic bands or cell thresholds instead of claiming exact cost;
- warn at yellow/red levels and permit intentional override;
- reject only invalid input (`v <= 0`, non-finite bounds) in an add-on;
- use saturating integer arithmetic to avoid overflow;
- debounce estimates while fields change and avoid copying/evaluating the stack on every panel draw.

All dimensions must use the same coordinate space as `v`. Combining world-scaled `Object.dimensions` with an object-space voxel size is precisely the mismatch this proposal should prevent.

## Preview feasibility

For the destructive Voxel Remesh operation, Blender already lets `R` adjust voxel size with an interactive grid overlay; `Shift` increases adjustment precision. The proposal should extend this interaction with cost annotations and warnings rather than describe interactive adjustment as entirely new.

An add-on can approximate the preview with a `SpaceView3D` draw handler and GPU line drawing. It should never draw every voxel. Cap grid slices or ticks, show a sample cell plus representative planes, annotate longest-axis count and risk band, and fall back to a bounding box at extreme density.

An add-on can reasonably:

- provide its own sidebar/popover, modal preview, and wrapper operators;
- read and write mesh-remesh and modifier RNA properties;
- cache local or evaluated bounds and risk scores;
- show a bounded viewport overlay;
- confirm risky executions while offering an explicit override.

Blender core is required or strongly preferred to:

- annotate the native Sculpt Remesh popover and Remesh modifier panel;
- extend the native `R` overlay;
- reliably cover native `Ctrl-R`, search/menu/keymap, and scripted invocation;
- expose accurate OpenVDB allocation information;
- enforce a shared confirmation policy in `OBJECT_OT_voxel_remesh`.

An add-on wrapper cannot guarantee safety when the native operator is invoked directly.

## Implementation sequence

1. Implement a pure estimator returning geometry source, coordinate space, bounds, voxel size, longest-axis cells, saturating bounding-cell score, surface score, scale warning, and confidence label.
2. Test tiny and huge objects, zero extents, empty meshes, unapplied uniform scale, non-uniform/negative scale, shear, and overflow.
3. Prototype an opt-in add-on panel and wrapper, keeping native values unchanged and thresholds configurable.
4. Add a capped viewport overlay; do not perform an actual remesh to generate the preview.
5. Calibrate risk bands on multiple Blender 5.1 builds, meshes, and hardware classes.
6. For upstream work, place shared estimation and warning policy in core while preserving the different geometry sources of the destructive operation and modifier.

## Verification caveats

This research was performed without a local Blender source checkout or executable, so live Blender 5.1 RNA introspection and performance measurements were not available. Before implementation:

- verify all defaults against the exact targeted Blender 5.1 build with `bl_rna.properties[...]`;
- confirm the operator's current geometry-source and transform behavior in source and with regression tests;
- inspect the evaluated-geometry boundary available to the Remesh modifier;
- calibrate thresholds empirically rather than presenting the formulas above as memory or face-count predictions.

The documentation URLs using `/dev/` and `/current/` are moving targets. Pin links to a Blender 5.1 documentation snapshot or source revision when this proposal moves from research to implementation.
